### 🎯 核心目标：保护本地开发代码

**我们的差异代码 = 本地开发代码 = 必须保护的代码**

```
上游代码 (openclaw/openclaw)  →  保存在 src/, ui/
    +
我们的本地开发代码           →  ⚠️ 这些必须移到 src-local/, ui-local/, i18n-local/
    =
当前的 JieZi AI-PS 代码库
```

**重构的本质：**
- ❌ 不是删除本地代码
- ❌ 不是覆盖本地修改
- ✅ 是「物理隔离」上游代码和本地代码
- ✅ 让 Git 合并时能区分哪些可以覆盖，哪些必须保留

**如何保证不丢失代码？**

1. **双重备份** - Git 分支 + 文件系统备份
2. **先复制后删除** - 永远不直接移动或删除
3. **逐步验证** - 每步都有检查点
4. **提取差异** - 使用 `git diff` 明确知道改了什么
5. **测试验证** - 构建和运行测试确认功能完整

### 3.3 创建 i18n-local/index.ts 合并层

**创建新文件** `i18n-local\index.ts`

```typescript
// i18n-local/index.ts
// 🎯 这个文件的作用：合并上游翻译和本地翻译

import { i18n as upstreamI18n, t as upstreamT } from '../ui/src/i18n/index.ts';
import type { Locale } from '../ui/src/i18n/lib/types.ts';

// 导入本地额外翻译（现在先创建空对象，后面再填充）
const zhCNExtra: any = {};
const zhTWExtra: any = {};
const ptBRExtra: any = {};

class LocalizedI18n {
  private locale: Locale = 'en';
  
  // 本地额外翻译（补充上游）
  private extraTranslations: Record<string, any> = {
    'zh-CN': zhCNExtra,
    'zh-TW': zhTWExtra,
    'pt-BR': ptBRExtra,
  };

  async setLocale(locale: Locale) {
    this.locale = locale;
    // 同步上游 i18n
    if (upstreamI18n?.setLocale) {
      await upstreamI18n.setLocale(locale);
    }
  }

  getLocale(): Locale {
    return this.locale;
  }

  t(key: string, params?: Record<string, any>): string {
    // 1. 优先使用本地额外翻译（本地功能的翻译）
    const localTranslation = this.getNestedValue(
      this.extraTranslations[this.locale], 
      key
    );
    if (localTranslation !== undefined) {
      return this.interpolate(localTranslation, params);
    }

    // 2. fallback 到上游翻译
    if (upstreamT) {
      return upstreamT(key, params);
    }

    // 3. 返回 key 本身
    return key;
  }

  private getNestedValue(obj: any, path: string): any {
    if (!obj) return undefined;
    const keys = path.split('.');
    let current = obj;
    for (const key of keys) {
      if (current?.[key] === undefined) return undefined;
      current = current[key];
    }
    return current;
  }

  private interpolate(text: string, params?: Record<string, any>): string {
    if (!params) return text;
    return text.replace(/\{(\w+)\}/g, (_, key) => params[key] || '');
  }
}

export const i18n = new LocalizedI18n();
export const t = i18n.t.bind(i18n);
export type { Locale };
```

## 🎓 概念解释（给新手）

### 什么是"三层架构"？

想象一个三明治：

```
🍞 第3层（i18n-local/）  - 国际化翻译层
🥬 第2层（src-local/, ui-local/） - 我们的本地功能
🍞 第1层（src/, ui/）    - 上游原始代码
```

每层独立存放，互不覆盖。

### 什么是"相对路径"？

```
项目结构：
project/
  ├── i18n-local/
  │   └── frontend/
  │       └── views/
  │           └── overview.ts  ← 你在这里
  └── ui/
      └── src/
          └── ui/
              └── views/
                  └── overview.ts  ← 你要引用这个

路径计算：
从 overview.ts（当前）到 overview.ts（目标）
1. ../ 上一级到 views/
2. ../ 上一级到 frontend/
3. ../ 上一级到 i18n-local/（项目根目录）
4. ui/src/ui/views/overview.ts

结果：../../../ui/src/ui/views/overview.ts
```

### 为什么要"恢复上游文件"？

因为我们直接修改了上游的文件（如 overview.ts），下次合并时会冲突。

解决方法：
1. 把我们的修改移到 `i18n-local/`（新位置）
2. 把上游文件恢复原样（让上游随便覆盖）
3. 应用入口改用我们在 `i18n-local/` 的版本

### 11.7 常见问题

**Q1: 每次启动都检查会不会很慢？**

A: 快速模式（--quick）只检查文件是否存在，通常 < 1秒
   完整模式（--full）包含编译检查，需要 5-10秒

**Q2: 检查失败了怎么办？**

A: 检查会给出具体的恢复建议，按照提示操作即可
   如果不确定，查看备份分支：git show backup-before-refactor:文件路径

**Q3: 可以跳过检查吗？**

A: 可以，但不建议
   临时跳过：pnpm dev --skip-check
   永久禁用：删除 package.json 中的 predev 钩子

## 📚 第12步：配置文档维护策略（10分钟）

**解决问题：我们的本地文档和上游文档如何分开维护**

### 为什么需要分离文档？

```
问题场景：
上游的 README.md 是英文
我们的 README-ZH.md 是中文，记录了本地功能

❌ 不分离：
   - 上游更新 README.md
   - 我们的 README-ZH.md 可能被覆盖
   - 或者合并时产生冲突

✅ 分离维护：
   - docs/upstream/ 放上游文档（只读）
   - docs/local/ 放本地文档（受保护）
   - 根目录 README.md 同时指向两者
```
